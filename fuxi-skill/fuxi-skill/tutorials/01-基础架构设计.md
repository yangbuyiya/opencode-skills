## 📝 关键内容

### 1. Phase流程设计

#### 1.1 为什么要分Phase？

**问题：直接让AI"生成一个乔布斯Skill"会怎样？**

❌ **结果：**
- 信息不完整（遗漏重要来源）
- 提炼浅薄（停留在表面引用）
- 质量不可控（没有验证机制）
- 不可复现（每次结果不同）

✅ **分Phase的好处：**
- 每个阶段有明确的目标
- 有质量检查点
- 可中断、可迭代
- 结果可复现

#### 1.2 5个核心Phase

```
Phase 0: 入口分流
  ↓
Phase 1: 多源信息采集（6个并行Agent）
  ↓
Phase 2: 框架提炼（三重验证）
  ↓
Phase 3: Skill构建（模板填充）
  ↓
Phase 4: 质量验证（3项测试）
```

#### 1.3 Phase 0: 入口分流

**目标：** 理解用户意图，选择正确的执行路径。

**两种路径：**

**路径A：直接路径**
```
用户输入：明确的人物名称
例如："帮我生成一个Naval的Skill"

↓
Phase 0A: 需求澄清
- 确认人物身份（避免重名）
- 确认类型（真实/历史/虚拟）
- 确认信息源偏好
- 确认时间范围

↓
Phase 1: 多源信息采集
```

**路径B：诊断路径**
```
用户输入：模糊的需求
例如："我想了解一个投资大师的思维"

↓
Phase 0B: 需求诊断
- 分析用户需求（投资、决策、创业？）
- 推荐候选人物（芒格、巴菲特、达利欧...）
- 用户选择

↓
Phase 0A: 需求澄清
↓
Phase 1: 多源信息采集
```

**Phase 0的决策树：**

```
用户输入
  ├─ 包含明确人名？
  │   ├─ 是 → Phase 0A（需求澄清）
  │   └─ 否 → Phase 0B（需求诊断）
  │
  └─ Phase 0A 输出
      ├─ 人物名称
      ├─ 人物类型
      ├─ 信息源偏好
      └─ 时间范围
```

#### 1.4 Phase 1: 多源信息采集

**目标：** 通过6个并行Agent，从不同维度采集信息。

**为什么是6个Agent？**

| Agent | 维度 | 为什么重要？ |
|-------|------|------------|
| 1 著作 | 深度思想 | 书籍是深思熟虑的表达 |
| 2 对话 | 即兴反应 | 播客/访谈展现真实思考 |
| 3 表达 | 语言特征 | 社交媒体展现日常风格 |
| 4 他者 | 外部视角 | 他人的观察补充盲点 |
| 5 决策 | 行为模式 | 实际行动验证思想 |
| 6 时间线 | 演变轨迹 | 思想不是静态的 |

**并行执行的意义：**
- 效率：6个Agent同时工作，节省时间
- 独立：每个Agent独立调研，互不干扰
- 多样：不同视角，交叉验证

**Phase 1的输出：**
```
references/research/
├── 01-writings.md      # Agent 1 的调研结果
├── 02-conversations.md # Agent 2 的调研结果
├── 03-expression-dna.md
├── 04-external-views.md
├── 05-decisions.md
└── 06-timeline.md
```

#### 1.5 Phase 2: 框架提炼

**目标：** 从原始信息中提炼心智模型、决策启发式、表达DNA。

**三重验证机制：**

一个观点要成为心智模型，必须通过：

1. **跨域复现**（Cross-domain Recurrence）
   - 在≥2个领域出现
   - 不是特定场景的偶然观点

2. **生成力**（Generative Power）
   - 能推断新问题的立场
   - 有解释力，不只是描述

3. **排他性**（Exclusivity）
   - 不是所有聪明人都这样想
   - 有辨识度，不是共识

**示例：验证"逆向思维"是否是芒格的心智模型**

```
验证1：跨域复现
- 投资：反向思考如何亏钱
- 生活：反向思考如何痛苦
- 决策：反向思考如何失败
✅ 通过（3个领域）

验证2：生成力
- 新问题：芒格会怎么看AI？
- 推断：他会反向思考"AI会如何失败"
✅ 通过（能推断新问题）

验证3：排他性
- 不是所有投资者都用逆向思维
- 这是芒格的独特方法
✅ 通过（有辨识度）

结论："逆向思维"是芒格的心智模型 ✅
```

**Phase 2的输出：**
```
references/extraction-framework.md
├── 心智模型（3-7个）
├── 决策启发式（5-10个）
├── 表达DNA
├── 反模式
├── 价值观
└── 智识谱系
```

#### 1.6 Phase 3: Skill构建

**目标：** 将提炼的框架填入skill-template，生成最终Skill。

**输入：**
- extraction-framework.md（提炼结果）
- skill-template.md（模板）

**输出：**
- SKILL.md（最终Skill）

**构建过程：**

```
extraction-framework.md
         ↓
    [模板填充]
         ↓
     SKILL.md
```

**关键规则：**
- 不要添加模板外内容
- 保持结构一致性
- 标注来源和可信度
- 包含诚实边界

#### 1.7 Phase 4: 质量验证

**目标：** 验证Skill是否真的像这个人。

**3项测试：**

**测试1：已知测试**
```
选择3个此人公开表态过的问题
让Skill回答
对比真实表态
评估一致性
```

**测试2：边缘测试**
```
选择1个此人未公开讨论的问题
让Skill推断
标注"这是推测"
验证是否合理
```

**测试3：风格测试**
```
让Skill生成100字表达
判断是否能识别出此人
评估表达DNA的辨识度
```

**通过标准：**

| 指标 | 标准 |
|-----|------|
| 已知测试一致性 | ≥80% |
| 边缘测试合理性 | 有推理逻辑，非胡编 |
| 风格测试辨识度 | 能识别核心特征 |
| 心智模型数量 | 3-7个 |
| 一手来源占比 | >50% |

### 2. 入口分流机制

#### 2.1 为什么需要入口分流？

**问题场景：**

❌ 用户输入："帮我生成一个投资大师的Skill"
- 哪个投资大师？芒格？巴菲特？索罗斯？
- AI瞎猜 → 用户不满意 → 浪费时间

❌ 用户输入："生成一个乔布斯的Skill"
- 哪个乔布斯？Steve Jobs？其他乔布斯？
- AI假设 → 可能出错 → 质量问题

✅ **入口分流的作用：**
- 明确用户意图
- 避免方向错误
- 提高成功率

#### 2.2 直接路径（Phase 0A）

**触发条件：** 用户输入包含明确的人物名称

**执行流程：**

```
用户输入："生成一个Naval的Skill"

↓
Step 1: 人物识别
- 搜索"Naval"
- 发现多个候选：
  1. Naval Ravikant（AngelList创始人）
  2. 其他Naval...

↓
Step 2: 需求澄清
AI: "我找到了多个Naval，你是指："
    "1. Naval Ravikant（AngelList创始人，天使投资人）"
    "2. 其他Naval"
    
用户: "第1个"

↓
Step 3: 类型确认
AI: "Naval Ravikant是真实在世人物，对吗？"
用户: "是的"

↓
Step 4: 信息源偏好
AI: "我会搜索以下信息源："
    "- 著作：《Navalmanack》、博客文章"
    "- 对话：播客、访谈"
    "- 表达：Twitter"
    "有需要调整的吗？"
用户: "没有"

↓
Step 5: 时间范围
AI: "我会包含截至当前的所有公开信息，"
    "重点更新最近12个月的动态。"
    "开始采集？"
用户: "开始"

↓
Phase 1: 多源信息采集
```

**Phase 0A的输出模板：**

```markdown
# Phase 0A 输出

## 人物信息
- 名称：Naval Ravikant
- 类型：真实在世人物
- 身份：AngelList创始人、天使投资人

## 信息源策略
- 著作：《Navalmanack》、博客
- 对话：播客（Joe Rogan、Tim Ferriss等）
- 表达：Twitter (@naval)
- 他者：他人分析、书评
- 决策：投资决策、创业决策
- 时间线：1990年代至今

## 时间范围
- 截止：当前
- 重点：最近12个月

## 下一步
进入 Phase 1: 多源信息采集
```

#### 2.3 诊断路径（Phase 0B）

**触发条件：** 用户输入不包含明确的人物名称

**执行流程：**

```
用户输入："我想学习一个投资大师的思维"

↓
Step 1: 需求分析
AI分析：
- 关键词：投资、大师、思维
- 意图：学习投资思维
- 可能候选：芒格、巴菲特、达利欧、索罗斯...

↓
Step 2: 候选推荐
AI: "我理解你想学习投资思维。以下是一些建议："
    
    "1. 查理·芒格 - 价值投资、逆向思维、多学科视角"
    "   适合：想学习决策框架和思维模型"
    
    "2. 瑞·达利欧 - 原则驱动、系统性思考"
    "   适合：想学习建立决策系统"
    
    "3. 沃伦·巴菲特 - 价值投资、长期主义"
    "   适合：想学习投资哲学"
    
    "4. 纳西姆·塔勒布 - 反脆弱、风险思维"
    "   适合：想学习应对不确定性"
    
    "你想选择哪一个？或者有其他候选人？"

用户: "查理·芒格"

↓
Step 3: 转入Phase 0A
（后续流程与直接路径相同）
```

**Phase 0B的输出模板：**

```markdown
# Phase 0B 输出

## 用户需求
- 原始输入："我想学习一个投资大师的思维"
- 解析意图：学习投资思维
- 推荐类型：投资大师

## 推荐候选
1. 查理·芒格（价值投资、逆向思维）
2. 瑞·达利欧（原则驱动、系统思考）
3. 沃伦·巴菲特（价值投资、长期主义）
4. 纳西姆·塔勒布（反脆弱、风险思维）

## 用户选择
- 选择：查理·芒格
- 原因：学习决策框架和思维模型

## 下一步
转入 Phase 0A: 需求澄清
```

#### 2.4 两种路径的对比

| 维度 | 直接路径 | 诊断路径 |
|-----|---------|---------|
| 触发条件 | 明确人名 | 模糊需求 |
| 执行步骤 | Phase 0A → Phase 1 | Phase 0B → Phase 0A → Phase 1 |
| 用户交互 | 确认和细化 | 选择和确认 |
| 适用场景 | 用户有明确目标 | 用户需要建议 |

### 3. 目录结构设计

#### 3.1 设计原则

**原则1：自包含（Self-contained）**
```
✅ 好：复制整个skill目录就能独立使用
❌ 坏：依赖外部文件，复制后无法使用
```

**原则2：层次清晰**
```
✅ 好：每层有明确职责
❌ 坏：文件乱放，难以维护
```

**原则3：可扩展**
```
✅ 好：新增内容有明确位置
❌ 坏：不知道该放哪里
```

#### 3.2 完整目录结构

```
.qclaw/skills/[person-name]-perspective/
│
├── SKILL.md                    # 最终Skill文件
│
├── README.md                   # 使用说明
│
├── LICENSE                     # 许可证
│
└── references/                 # 参考资料目录
    │
    ├── skill-template.md       # Skill模板
    │
    ├── extraction-framework.md # 提炼方法论
    │
    ├── research/               # Agent调研结果
    │   ├── 01-writings.md      # Agent 1: 著作
    │   ├── 02-conversations.md # Agent 2: 对话
    │   ├── 03-expression-dna.md
    │   ├── 04-external-views.md
    │   ├── 05-decisions.md
    │   └── 06-timeline.md
    │
    └── sources/                # 原始资料（可选）
        ├── books/              # 书籍PDF/摘录
        ├── transcripts/        # 访谈转录
        └── articles/           # 文章备份
```

#### 3.3 各目录职责

**SKILL.md**
- 最终交付物
- 可直接使用
- 包含完整的人设、心智模型、表达DNA

**README.md**
- 使用说明
- 安装方法
- 效果示例
- 注意事项

**LICENSE**
- 许可证
- 开源协议（MIT/Apache等）

**references/skill-template.md**
- Skill的模板结构
- 定义各section的格式
- 确保一致性

**references/extraction-framework.md**
- 提炼方法论
- 三重验证详解
- 质量标准

**references/research/**
- 6个Agent的调研结果
- 原始信息记录
- 提炼的素材来源

**references/sources/**（可选）
- 原始资料备份
- 书籍、访谈、文章
- 用于验证和追溯

#### 3.4 文件命名规范

**调研文件命名：**
```
格式：0X-[dimension].md
示例：01-writings.md, 02-conversations.md

规则：
- 前缀数字表示Agent编号
- 英文短横线分隔
- 全小写
```

**source文件命名：**
```
格式：[YYYY-MM-DD]-[source-name].[ext]
示例：2023-05-15-naval-podcast-transcript.txt

规则：
- 日期前缀（可选）
- 来源名称
- 文件扩展名
```

### 4. 检查点设计

#### 4.1 为什么需要检查点？

**问题：没有检查点会怎样？**

❌ 场景1：
- Phase 1执行完，发现信息不足
- 需要重新采集
- 浪费时间和资源

❌ 场景2：
- Phase 2提炼完，发现质量不够
- 需要重新提炼
- 前功尽弃

✅ **检查点的作用：**
- 及早发现问题
- 节省迭代成本
- 保证质量

#### 4.2 关键检查点

```
Phase 0 → Phase 0.5 检查 → Phase 1
Phase 1 → Phase 1.5 检查 → Phase 2
Phase 2 → Phase 2.5 检查 → Phase 3
Phase 3 → Phase 4 → 最终检查
```

#### 4.3 Phase 0.5：目录创建检查

**触发时机：** Phase 0完成后，Phase 1开始前

**检查内容：**
```markdown
# Phase 0.5 检查清单

## 目录结构
- [ ] 创建了主目录：[person-name]-perspective/
- [ ] 创建了references/目录
- [ ] 创建了references/research/目录
- [ ] 创建了references/sources/目录（可选）

## 文件准备
- [ ] 准备了skill-template.md（或使用默认模板）
- [ ] 准备了extraction-framework.md（或使用默认方法）

## Phase 0 输出
- [ ] Phase 0A或Phase 0B的输出文档已生成
- [ ] 人物信息已确认
- [ ] 信息源策略已明确

## 下一步
如果全部✅，进入Phase 1
如果有❌，先完成缺失项
```

#### 4.4 Phase 1.5：调研质量检查

**触发时机：** Phase 1完成后，Phase 2开始前

**检查内容：**
```markdown
# Phase 1.5 检查清单

## 调研文件完整性
- [ ] 01-writings.md 存在且不为空
- [ ] 02-conversations.md 存在且不为空
- [ ] 03-expression-dna.md 存在且不为空
- [ ] 04-external-views.md 存在且不为空
- [ ] 05-decisions.md 存在且不为空
- [ ] 06-timeline.md 存在且不为空

## 信息源质量
- [ ] 每个文件包含信息来源
- [ ] 来源可信度已标注
- [ ] 区分了一手/二手/推断

## 信息数量
- [ ] 总信息条目 ≥ 30条
- [ ] 每个维度 ≥ 5条
- [ ] 包含时间信息

## 信息质量
- [ ] 有矛盾信息已保留
- [ ] 有时间标注（避免过时信息）
- [ ] 有来源链接（可追溯）

## 下一步
如果全部✅，进入Phase 2
如果信息不足❌，补充调研
```

#### 4.5 Phase 2.5：提炼结果检查

**触发时机：** Phase 2完成后，Phase 3开始前

**检查内容：**
```markdown
# Phase 2.5 检查清单

## 心智模型
- [ ] 数量：3-7个
- [ ] 每个模型有三重验证
- [ ] 每个模型有应用场景
- [ ] 每个模型有局限性说明

## 决策启发式
- [ ] 数量：5-10个
- [ ] 每个启发式有具体场景
- [ ] 每个启发式有行为示例

## 表达DNA
- [ ] 有高频词汇
- [ ] 有句式特点
- [ ] 有修辞手法
- [ ] 有情绪基调
- [ ] 有典型句式示例

## 反模式
- [ ] 数量：3-5个
- [ ] 每个有证据支持

## 诚实边界
- [ ] 明确了时间边界
- [ ] 明确了信息边界
- [ ] 明确了能力边界

## 下一步
如果全部✅，进入Phase 3
如果质量不够❌，重新提炼或补充调研
```

#### 4.6 Phase 4：最终质量检查

**触发时机：** Phase 4完成后

**检查内容：**
```markdown
# Phase 4 最终检查清单

## 已知测试
- [ ] 选择了3个已知问题
- [ ] Skill回答与真实表态一致性 ≥ 80%
- [ ] 不一致的地方有合理解释

## 边缘测试
- [ ] 选择了1个边缘问题
- [ ] Skill有推理逻辑
- [ ] 标注了"这是推测"

## 风格测试
- [ ] 生成了100字表达
- [ ] 能识别出此人的核心特征
- [ ] 表达DNA有辨识度

## 结构完整性
- [ ] SKILL.md结构完整
- [ ] 包含所有必需section
- [ ] 格式符合模板

## 来源标注
- [ ] 一手来源占比 > 50%
- [ ] 所有重要观点有来源
- [ ] 推测部分已标注

## 下一步
如果全部✅，Skill完成
如果质量不够❌，迭代优化
```

## ❓ 常见问题

### Q1: Phase必须按顺序执行吗？

**A:** 是的，有依赖关系：

- Phase 1依赖Phase 0的输出
- Phase 2依赖Phase 1的调研
- Phase 3依赖Phase 2的提炼
- Phase 4依赖Phase 3的Skill

但可以在某个Phase失败后，退回重试。

### Q2: 检查点失败怎么办？

**A:** 根据失败原因处理：

- 信息不足 → 补充调研
- 质量不够 → 重新提炼
- 结构不对 → 修正模板

不要跳过检查点，否则后期问题更大。

### Q3: sources目录是必需的吗？

**A:** 不是必需，但建议有：

**有sources的好处：**
- 可追溯来源
- 可验证信息
- 可更新资料

**没有sources的场景：**
- 信息源太多（如Twitter全量归档）
- 版权限制
- 存储空间限制

### Q4: 可以跳过Phase 0吗？

**A:** 如果用户输入非常明确，可以简化Phase 0：

```
用户输入："生成Naval Ravikant的Skill，包含他的书和播客"
↓
跳过Phase 0，直接开始Phase 1
```

但建议至少确认人物类型和信息源。

### Q5: 6个Agent都必须运行吗？

**A:** 根据人物类型调整：

**真实在世人物：** 6个全运行
**历史人物：** 重点运行1、4、6，其他降级
**虚拟人物：** 重点运行1、5、6，其他降级

信息源不足时，要调整预期。
